Reflection

The hardest part of this project was not one single bug, but the fact that the work had to hold together across three different stages. The downloader, the cleaning step and the Spark analysis all depended on one another, so if something went wrong early on it would carry through the rest of the pipeline. At first I treated the tasks as separate exercises, but it quickly became clear that the real challenge was making the whole workflow consistent. I had to think about whether the data being downloaded was reliable, whether the cleaning decisions made sense, and whether the final analysis would be trustworthy.

What helped most was slowing down and checking each step more carefully. Instead of changing lots of things at once, I worked through the pipeline stage by stage. For the downloader, that meant verifying the API responses and making sure the rate limiter was actually controlling traffic. I also checked that the CSV output had the expected structure and the expected number of rows. For the cleaning stage, I inspected the data before changing it and then reviewed the result afterwards so I could tell whether the transformations had worked. Writing the logic out in plain English first also helped. It made the code easier to reason about and made it easier to spot when an assumption was wrong.

Multithreading did make the API download much faster, but it also made the project more complicated. The difference was very clear in the benchmark: the serial download took about 17.18 seconds, while the multithreaded version finished in about 2.11 seconds. That is a major improvement, especially because the script was mostly waiting on network responses rather than doing heavy computation. At the same time, concurrency introduced real risks. If several threads wrote to the log file at once, the output could become messy and harder to use. That is why the rate limiter and the lock around the log writer mattered. The downloader was not just about speed; it was about making sure the process stayed controlled and readable while running in parallel. In that sense, multithreading improved the workflow, but only because I built safeguards around it.

Pandas was the right tool for the cleaning stage because it made the work feel practical and interactive. It was useful for looking at the messy file, checking which columns had missing values, and seeing which fields were being read incorrectly. In this project, the data had been deliberately corrupted, so I had to decide how to handle a mix of missing values, bad timestamps, text in numeric columns, inconsistent symbols, duplicates and a few impossible values. Pandas made it straightforward to convert columns safely, remove duplicates, normalise symbols, and create new fields such as price range and candle direction. It also helped me decide which rows were too unreliable to keep. In the end, the cleaned dataset went from 9,991 rows to 9,628 rows, which felt reasonable because some rows clearly did not contain trustworthy information. I did not try to fill everything in artificially; I focused on keeping the useful rows and removing the ones that would distort the analysis.

Spark was a better fit for the final analysis. Once the data had been cleaned, it made sense to use a tool designed for bigger, grouped computations rather than just a small sample. The pandas work was limited to 50 rows as a quick check, but Spark allowed me to work with the full cleaned dataset and calculate more meaningful summaries across all the symbols. I used it to create a temporary SQL view, run grouped queries, and generate a ranked market summary. That part of the project was much more useful from a business point of view because it moved the work beyond “does the data look okay?” to “what does the data tell us?” The results showed clear differences between the symbols, especially in activity and volatility. BTCUSDT stood out as the most active and volatile symbol in the summary, while other coins showed lower levels of trading activity and narrower price movement.

One thing this project made clear is that data cleaning is not just a mechanical step. It involves judgement. There is not always a single obvious answer when a dataset has been damaged or corrupted. I had to decide whether a value should be corrected, dropped, or treated as missing, and those choices directly affected the final analysis. Before this project, I thought cleaning was mostly about making the data “look tidy.” I now see that it is a much more important part of the workflow because the cleaning decisions shape the conclusions later on.

If I had more time, I would make the pipeline a bit more robust in a few ways. I would add stronger validation checks between each stage so the process could stop earlier if something looked wrong. I would also try to automate the full workflow so one command could run the download, cleaning and Spark summary steps together. On top of that, I would improve the API handling with retries and clearer error messages so that temporary failures would be less disruptive. Those changes would make the project easier to reuse and more reliable in practice.

Overall, this project helped me understand how the different parts of a data pipeline fit together. Writing the code was only one part of it. The other part was making sure the data was trustworthy, the outputs were verified, and the final analysis was built on choices that made sense. Pandas and Spark each had a clear role. Pandas was useful for inspection and preparation, while Spark was better for the full analysis. Looking back, the biggest lesson was that it is worth taking the time to build confidence in each stage before moving on to the next one.
