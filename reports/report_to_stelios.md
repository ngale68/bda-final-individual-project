Report to Stelios
I was asked to build a small end-to-end market data pipeline for Owl Analytics, covering data collection, cleaning and analysis. The goal was to turn raw data from the Binance API into something reliable enough to support a clear set of summary analytics. Rather than going through every implementation detail, this report focuses on how the pipeline works, how data quality was improved, and what the final analysis showed.

The pipeline starts by downloading hourly candlestick data for ten cryptocurrency pairs: BTCUSDT, ETHUSDT, BNBUSDT, SOLUSDT, XRPUSDT, ADAUSDT, DOGEUSDT, AVAXUSDT, LINKUSDT and DOTUSDT. Each request returns up to 1,000 records, so the target was a dataset of around 10k rows. I wrote the downloader in Python to request the data, map the JSON response into a consistent CSV structure and save the results to data/clean/clean_market_data.csv.

The main challenge in this part was getting the data quickly without breaking Binance’s rate limits. The API is public, but it still restricts how fast requests can be made. To stay within those limits, I added a rate limiter that allowed a maximum of 100 requests per minute while still letting several symbols download at the same time. I also used a ThreadPoolExecutor so the script could send multiple requests concurrently instead of waiting for one symbol to finish before moving on to the next. That made a big difference in practice. The serial version took about 17.18 seconds, while the multithreaded version finished in roughly 2.11 seconds.

Running requests in parallel brought one more issue: several threads could try to write to the same log file at the same time. Without protection, the log entries could get mixed up and become harder to read. I fixed that by wrapping each log write in a lock so only one thread could write at a time. The log records when each request starts and finishes, how many rows were returned for each symbol, and when the final CSV file was written. That made it much easier to verify the process and to troubleshoot anything that went wrong.

After the collection stage, the next step was cleaning the deliberately messy dataset produced by Dara. The raw data had several common quality issues, including missing values, text stored in numeric columns, invalid timestamps, inconsistent symbol names, duplicate rows and impossible values such as negative trading volumes. These are the sort of problems that often appear in real-world data, so they had to be handled before any meaningful analysis could be done.

I used pandas for the cleaning stage because it is well suited to inspecting and transforming tabular data. I started by looking closely at the dataset to understand the different types of corruption before deciding how to fix them. Numeric columns were converted safely, so invalid text became missing values instead of breaking the script. Timestamp fields were parsed where possible, and invalid dates were flagged and treated as missing. Symbol names were standardised so variations like btcusdt, BTCUSDT and BTC/USDT all became the same value. Duplicate rows were removed, and rows with impossible values such as negative trading volumes or invalid price relationships were discarded.

The cleaning process reduced the dataset from 9,991 rows to 9,628 rows. Most of the removed records were simply too unreliable to keep. I considered filling in missing values, but because the corruption had been introduced deliberately for the exercise, removing the bad rows was the safer and more defensible approach. The aim was not to maximise the number of rows, but to create a dataset that could be analysed with confidence. The quality report showed that the biggest issues were missing timestamps and missing price-related values, both of which were addressed in the cleaning step.

Pandas was used only on a 50-record sample because the assignment separates exploratory cleaning from the full analysis. It is useful for quickly inspecting data, testing cleaning logic and checking data types on a small subset. Zehra then asked us to use Spark for the full analysis because it is built for larger datasets and handles grouped aggregations and SQL-style analytics much more effectively. Even though this project used fewer than 10,000 rows, the same Spark workflow would scale much more easily to larger datasets.

The final Spark analysis showed some clear differences between the cryptocurrency pairs. BTCUSDT recorded the highest total trade count and the widest average price range, which suggests it was both the most actively traded and the most volatile asset in the dataset. ETHUSDT also performed strongly across several metrics, while a few smaller-cap tokens had lower trading volumes and narrower price movements. Spark produced summary statistics for each symbol, including total trading volume, trade count, average percentage price change, average price range and the balance of bullish and bearish candles. Ranking the symbols across these measures made it much easier to compare their behaviour and identify the most active markets.

Overall, the pipeline met its objective of collecting, cleaning and analysing cryptocurrency market data. The downloader produced a consistent dataset while staying within the API’s rate limits, the cleaning stage removed unreliable records without introducing artificial values, and the Spark analysis generated clear summary statistics across all ten pairs. Taken together, those steps produced a dataset that was reliable enough for further analysis and showed a workflow that could be extended to much larger market datasets.
